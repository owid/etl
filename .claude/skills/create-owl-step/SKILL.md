# Create Owl Step

Owl is a lightweight pipeline runner for **small** OWID datasets. Instead of separate snapshot, meadow, garden, grapher steps plus DAG edits, one Owl step folder holds everything and writes a normal `owid.catalog` dataset under `data/garden/`.

Use Owl for: prototypes, demos, tiny single-source datasets, agent-assisted dataset creation. **Do not** use Owl for mature, high-traffic, multi-stage datasets — full ETL is still the safer default there. If the dataset is large or needs real DAG integration, stop and recommend the classic chain (`/create-etl-steps`) instead.

The framework lives in `lib/owl/owl/`; the authoritative reference is `lib/owl/owl/README.md`. Steps live in `owl_steps/`.

## Layout

```txt
owl_steps/<namespace>/<dataset>/vYYYYMMDD/
  step.py             # @Snapshot / @Dataset / @Action functions
  meta.yml            # human-authored: snapshot origin + dataset/column metadata
  snapshot.lock.yml   # GENERATED by `owl snapshot` — content identity (md5/size/suffix). Never hand-edit.
  <dataset>.countries.json   # optional, for country harmonization
```

- `vYYYYMMDD` (Python-friendly) maps to the ETL version `YYYY-MM-DD`.
- Output: `data/garden/<namespace>/YYYY-MM-DD/<dataset>/` (a real catalog dataset).
- Raw snapshots cache to `data/snapshots/by-md5/<md5><suffix>`.

## Commands (always `.venv/bin/owl`)

```bash
.venv/bin/owl new <namespace>/<dataset> [--date YYYY-MM-DD]   # scaffold step.py + meta.yml
.venv/bin/owl snapshot <namespace>/<dataset>                  # fetch raw data, write snapshot.lock.yml
.venv/bin/owl run <namespace>/<dataset>                       # build catalog dataset (rebuilds if stale)
.venv/bin/owl run <namespace>/<dataset> --force               # rebuild even if up to date
.venv/bin/owl run <namespace>/<dataset> --grapher             # run grapher upsert actions
.venv/bin/owl run                                             # run all steps
.venv/bin/owl viz <namespace>/<dataset>                       # render the dependency DAG
```

The `<namespace>/<dataset>` argument is a **regex pattern** — partial matches work. `snapshot` is the *only* way to fetch data; `run` never fetches.

## Workflow

1. **Confirm Owl is the right tool.** Small/prototype dataset, single source file → yes. Otherwise recommend classic ETL.
2. **Scaffold:** `.venv/bin/owl new <namespace>/<dataset>` (namespace + dataset must be `[a-z0-9_]+`). Add `--date` to match the source's access date if known. This writes a runnable starter `step.py` + `meta.yml`.
3. **Edit `step.py`** — fill in the `@Snapshot` (how to fetch) and `@Dataset` (how to transform). See patterns below.
4. **Edit `meta.yml`** — fill the snapshot `origin` and dataset/column metadata. See metadata section.
5. **Fetch:** `.venv/bin/owl snapshot <namespace>/<dataset>` → writes `snapshot.lock.yml`.
6. **Build:** `.venv/bin/owl run <namespace>/<dataset>` → writes the catalog dataset. Run a second time to confirm it reports up to date.
7. Verify the output with a quick read of `data/garden/.../index.json` or by loading the table.

## Writing `step.py`

A step is plain Python with three decorators imported from `owl`. Dependencies are wired **by parameter name** — a parameter named `raw_data` resolves to the `@Snapshot`/`@Dataset` object of that name in the module.

### Recommended pattern (catalog Table + origins) — use this for real datasets

This mirrors the merged Karlinsky port (PR #6187) and preserves origins/metadata properly:

```python
from pathlib import Path

from owid.catalog import Table
from owl import Action, Dataset, Snapshot, SnapshotCapture
from owl.catalog import load_snapshot
from owl.grapher import upsert_dataset

from etl.data_helpers import geo

URL_DOWNLOAD = "https://example.com/data.csv"
COUNTRIES_FILE = Path(__file__).with_name("my_dataset.countries.json")


@Snapshot
def raw_data(snap: SnapshotCapture) -> None:
    snap.download(URL_DOWNLOAD, suffix=".csv")


@Dataset
def my_dataset(raw_data: Snapshot) -> Table:
    tb = load_snapshot(raw_data)            # Table with origins attached from meta.yml `origin`

    tb = tb.rename(columns={"country_name": "country"})
    tb = geo.harmonize_countries(df=tb, countries_file=COUNTRIES_FILE)
    tb = tb.format(["country", "year"])     # set index; Owl writes it as a catalog table
    return tb


@Action(kind="grapher", default=False)
def upsert_to_grapher(my_dataset: Dataset) -> None:
    upsert_dataset(my_dataset)
```

- `load_snapshot(raw_data)` reads the cached file via `owid.catalog.processing.read(...)` and attaches `origin` from `meta.yml` to every column. Pass reader kwargs through (e.g. `load_snapshot(raw_data, sep=";")`).
- Follow the repo's metadata-preserving rules (use `pr.*` not `pd.*`, no `np.where`, etc. — see CLAUDE.md "Preserving metadata/origins").
- Return either a `Table`/`DataFrame`, or `(df, DatasetMeta(...))`. When you return a Table and provide metadata in `meta.yml`, that YAML wins (deep-merged on top).

### Lightweight pattern (inline metadata, no YAML)

The `owl new` scaffold uses this — fine for quick demos:

```python
import pandas as pd

from owl import ColumnMeta, Dataset, DatasetMeta, Snapshot, SnapshotCapture


@Snapshot
def raw_data(snap: SnapshotCapture) -> pd.DataFrame:
    snap.download("https://example.com/data.csv")   # or: return pd.DataFrame(...)


@Dataset
def my_dataset(raw_data: Snapshot) -> tuple[pd.DataFrame, DatasetMeta]:
    df = raw_data.read_csv()
    return df, DatasetMeta(
        title="Dataset title",
        description="Dataset description",
        columns={
            "country": ColumnMeta(title="Country", role="entity"),
            "year": ColumnMeta(title="Year", role="time"),
            "value": ColumnMeta(title="Value", unit="..."),
        },
    )
```

### Depending on an existing ETL dataset (read-only)

```python
from owl import Dataset, ETLDataset

regions = ETLDataset("regions/2023-01-01/regions")   # defaults to garden channel

@Dataset
def my_dataset(regions: ETLDataset) -> Table:
    tb_regions = regions.read("regions")             # reset_index=True, safe_types=False by default
    ...
```

## `@Snapshot` capture API

The function receives a `SnapshotCapture` and must capture **exactly one file** (bundle multiple files into a zip):

- `snap.download(url, suffix=".csv")` — download a URL.
- `snap.add(path, suffix=...)` — register an already-downloaded file.
- `snap.write_bytes(data, suffix=...)` / `snap.write_text(data, suffix=".txt")` — write generated content.
- Or `return` a `pandas.DataFrame` (stored as parquet) / a file path.

Load it back in the dataset with `load_snapshot(snap)` (preferred, attaches origins), or `snap.read_csv()` / `snap.read_excel()` / `snap.path()`.

## Writing `meta.yml`

Two top-level keys. `snapshot.lock.yml` is separate and **generated** — keep data-identity churn out of `meta.yml`.

```yaml
snapshots:
  raw_data:                      # must match the @Snapshot function name
    origin:
      producer: ...
      title: ...
      description: |-
        ...
      citation_full: ...
      url_main: ...
      url_download: ...
      date_accessed: 'YYYY-MM-DD'
      date_published: 'YYYY-MM-DD'
      license:
        name: CC BY 4.0
        url: https://creativecommons.org/licenses/by/4.0/

datasets:
  my_dataset:                    # must match the @Dataset function name
    definitions:
      common:
        presentation:
          topic_tags:
            - Causes of Death
    tables:
      my_dataset:
        variables:
          some_column:
            title: ...
            unit: ...
            description_short: |-
              ...
            display:
              numDecimalPlaces: 0
            presentation:
              title_public: ...
              attribution_short: ...
```

- The `datasets.<name>` block accepts the **standard catalog metadata shape** (`definitions`, `tables` → `variables`), exactly like a classic garden `.meta.yml`. When present it takes precedence and the inline `DatasetMeta` is not needed.
- Snapshot `origin` flows onto columns via `load_snapshot(...)`, and Owl falls back to the origin's title/description for the dataset if none is set.

## Notes & limitations

- Experimental API; limited metadata validation; no full DAG integration.
- One file per snapshot by design.
- Staleness is mtime-based on `step.py`, `meta.yml`, and dependency snapshots/datasets — editing any of these triggers a rebuild on the next `owl run` (no `--force` needed).
- Grapher actions only run with `--grapher` (or `--action grapher`); they're `default=False`.
- After editing, run `make check` before committing (per repo rules). Don't commit/push unless explicitly asked.
